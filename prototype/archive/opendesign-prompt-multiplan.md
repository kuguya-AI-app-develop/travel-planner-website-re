# Travel Planner 多计划功能 - OpenDesign Prompt

我在做一个旅行策划网站的 UI 原型，目前是一个单页面 HTML 文件，包含以下模块：行程日历（可拖拽）、机票对比、目的地选择（评分表）、酒店评分、其他消费、预算总结。

当前问题：所有数据都是全局的，只能管理一个旅行计划。但实际用户可能需要同时制定多个计划（比如"如果能请到假就去东京5天" vs "请不到假就在京都过周末"），然后根据实际情况选择执行哪个。

请帮我设计"多计划"功能的 UI，要求如下：

1. 计划选择器：在侧边栏顶部增加一个计划切换组件，显示当前计划名称，点击展开计划列表，可以快速切换。每个计划显示名称、状态标签、预算摘要。

2. 计划总览页：新增一个"计划总览"页面（放在侧边栏导航中），以卡片形式展示所有计划。每张卡片显示计划名称、目的地、日期范围、总预算、待办完成进度。支持勾选多个计划进行关键指标并排对比（总预算、天数、目的地）。

3. 计划管理：支持新建空白计划、从现有计划复制、重命名、删除。每个计划可标记状态（草稿/进行中/已确认/已出行/已完成），状态用不同颜色的标签显示。

4. 每个计划独立拥有封面图片、行程、机票、目的地、酒店、消费、待办清单数据。切换计划时所有模块数据自动刷新。

5. 保持现有的设计风格：oklch 色彩体系、Inter 字体、圆角卡片、左侧固定侧边栏、响应式布局（桌面/平板/手机三档）。现有的模块结构（行程日历、机票对比等）不需要改动内部布局，只需要支持多计划数据隔离。

6. 移动端适配：计划选择器在移动端应该是一个顶部的下拉/抽屉，计划总览卡片在手机上改为单列纵向列表。

7. 数据持久化（localStorage）：所有数据必须通过 localStorage 持久化存储，刷新页面后不丢失。具体要求：
   - 数据结构：以 plans 对象为顶层，key 为 planId，每个计划包含完整的独立数据（trips、flights、flightCriteria、destinations、destCriteria、hotels、hotelCriteria、expenses、checklistItems、nextId 计数器、封面图片、状态标签等）。全局维护一个 activePlanId 记录当前选中的计划。
   - 存储策略：使用一个统一的 localStorage key（如 'travel-planner-data'），将整个 plans 对象和 activePlanId 序列化为 JSON 存储。封面图片使用独立的 key 存储（按 planId 区分，如 'tp-cover-{planId}'），避免单个 localStorage entry 过大。
   - 保存时机：所有数据修改操作（增删改、勾选、评分、拖拽、重命名维度、切换计划、编辑 contenteditable 字段）都应自动触发保存。使用全局事件监听（blur/change）兜底，确保不遗漏。
   - 加载时机：页面初始化时从 localStorage 恢复数据，恢复成功后显示轻提示（如"已从本地恢复数据"）。如果没有存储数据则使用默认示例数据。
   - 重置功能：在侧边栏提供"重置数据"入口，确认后清除所有 localStorage 数据并刷新页面回到默认状态。
   - 兼容性：loadData 时对数据结构做防御性检查，确保旧版数据（单计划格式）能平滑迁移到新的多计划结构（自动包裹为一个默认计划）。

请给出完整的 HTML+CSS+JS 原型文件，包含示例数据（2-3个不同状态的计划），可以直接在浏览器打开运行。所有数据变更必须实时持久化到 localStorage。
